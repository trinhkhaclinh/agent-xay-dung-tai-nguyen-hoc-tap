3 cổng kiểm chất lượng (đã triển khai)

① Cổng SSOT — ngay sau Bước 1, trước khi fan-out (shift-left): qa_spec.py (đủ khóa, mỗi objective có nhãn Bloom, cả session ≥3 bậc Bloom, prior_session) + verify_code.py --run (chạy thử app FastAPI trong code_examples). Sửa SSOT một lần, rẻ hơn sửa 6 artifact.

② Bước 4 · Lớp A — QA tự động (cổng cứng): verify_code.py --run (cú pháp Python + smoke‑test FastAPI; file exercise_/quiz_ chỉ kiểm cú pháp vì cố tình chứa code lỗi) + qa_vault.py (frontmatter, wikilink, MOC, concept). Cả hai phải exit 0.

③ Bước 4 · Lớp B — Review bàn giao (bắt buộc): agent chuyen-gia-noi-dung-fastapi bắt phần tool không thấy — nhất quán bài đọc↔slide↔video, độ phủ backwards design (mỗi objective được DẠY và KIỂM), chất lượng quiz, lỗi logic của code (chạy được nhưng dạy sai).

Mọi FAIL/BLOCKER đều quay về sửa JSON trong _spec/ rồi render lại — không sửa thẳng .md (sẽ bị render ghi đè).

Chống "drift" khi fan-out song song

6 subagent chạy độc lập dễ bịa ví dụ/tên file khác nhau cho cùng một Lesson. Quy tắc cứng đã đưa vào các skill: code, tên file (main.py/gateway.py), scenario và số liệu dùng chung phải trích lại nguyên văn từ SSOT. SSOT là hợp đồng — artifact chỉ diễn giải, không phát minh dữ kiện mới. Nhờ đó cả 6 nhóm artifact khớp nhau ngay từ đầu, giảm rework ở cổng cuối.

Điểm gate đầu vào

Ngay sau khi đọc bối cảnh: nếu hinh_thuc_hoc ≠ "Lý thuyết" thì dừng và hỏi người dùng — pipeline hiện tối ưu cho session Lý thuyết. Thực hành / Mini Project / Hackathon / Đồ án thuộc đợt phát triển sau.

Còn lại (khi scale 30 session)

  • Build tăng tiến: QA báo lỗi theo từng artifact → chỉ render lại file đó thay vì cả vault.
  • Coherence xuyên session: quiz Đầu giờ của session N ôn lại N‑1 — cần đảm bảo nội dung N‑1 đã chốt.
  • Định nghĩa concept "đóng băng" ở session đầu giới thiệu (session sau chỉ thêm backref) — nhớ khi muốn tinh chỉnh.
